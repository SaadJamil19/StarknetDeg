1. Executive Summary
This document presents a comprehensive on-chain verification of the Degenter Starknet DEX indexer database schema. Each table has been verified by cross-referencing raw Starknet blockchain event data against indexed records. Additionally, missing fields have been identified, analysed, and prioritised with detailed justifications.
Tables verified in this report:
trades — DEX swap trade records (Ekubo / AVNU)
price_ticks — Per-event price snapshots
prices — Token price registry
pool_latest — Real-time pool state snapshots
ohlcv_1m — 1-minute OHLCV candlestick data
token_metadata — Token information registry
stark_transfers — ERC-20 token transfer records
stark_token_concentration — Token holder concentration (currently empty)
stark_message_l2_to_l1 — L2 to L1 cross-chain messages (currently empty)

8.3 Missing Fields — Detailed Analysis
9. stark_token_concentration & stark_message_l2_to_l1
9.1 stark_token_concentration
  CURRENTLY EMPTY — Low priority for Degenter DEX analytics phase.
This table tracks token holder distribution metrics: top-N holder percentages, whale counts, Gini coefficient, Herfindahl index. It is empty because the indexer processes Swap events, not Transfer events. Populating this table requires:
Indexing all ERC-20 Transfer events for tracked tokens
Maintaining a token_holders balance table (address, holder_address, balance_raw)
Running periodic concentration computations (Gini, top-N percentages)
Recommendation: Defer to Phase 3. For current DEX analytics use case, this data is available from CoinGecko / Starkscan API and can be cached rather than computed independently.
If implemented, key fields to add: token_address, block_number, block_timestamp, holder_count, top1_pct, top10_pct, top100_pct, whale_count, whale_pct_combined, herfindahl_index, gini_coefficient, snapshot_kind.
9.2 stark_message_l2_to_l1
  CURRENTLY EMPTY — Only populated by bridge withdrawals and cross-chain governance.
L2-to-L1 messages are emitted by Starknet contracts calling the send_message_to_l1 syscall. Primary use case is StarkGate bridge withdrawals. The table is empty because the indexer currently listens to DEX contract events (Ekubo, AVNU), not Starknet core contract events (address 0x1).
To populate: listen to MessageSentToL1 events from contract 0x0000...0001 (Starknet core). Parse from_address (L2 sender), to_address (L1 recipient), payload array. For StarkGate bridge messages, decode payload[0]=l1_recipient, payload[1..2]=amount (uint256 low/high bits).
Key fields to add: message_hash, from_address, to_address, payload (JSONB), status (SENT/CONSUMED/CANCELLED), consumed_at_block, consumed_at_tx, is_bridge_withdrawal, bridge_token_address, bridge_l1_recipient, bridge_amount_raw, bridge_amount_usd, bucket_1m.
10. Recommended New Table: tokens
A centralised token registry table is recommended as a foundation for all other tables. Multiple tables currently embed token information redundantly (symbol, decimals, name in stark_transfers metadata; decimals needed in trades, price_ticks, prices; verification status in multiple places).
10.1 Recommended Schema
CREATE TABLE public.tokens (
  address              TEXT PRIMARY KEY,
  symbol               TEXT NOT NULL,          -- ETH, USDC, STRK
  name                 TEXT NOT NULL,          -- Ether, USD Coin
  decimals             INTEGER NOT NULL,       -- 18, 6, 8
  token_type           TEXT DEFAULT 'ERC20',
  is_stable            BOOLEAN DEFAULT FALSE,  -- USDC/USDT/DAI
  is_verified          BOOLEAN DEFAULT FALSE,
  coingecko_id         TEXT,
  logo_url             TEXT,
  deploy_tx_hash       TEXT,
  deployed_at          TIMESTAMPTZ,
  metadata             JSONB,
  created_at           TIMESTAMPTZ DEFAULT NOW(),
  updated_at           TIMESTAMPTZ DEFAULT NOW()
);

10.3 Readable View — trades_readable
CREATE VIEW trades_readable AS
SELECT t.*,
  t_in.symbol      AS token_in_symbol,
  t_in.decimals    AS token_in_decimals,
  t_out.symbol     AS token_out_symbol,
  t_out.decimals   AS token_out_decimals,
  (t.amount_in::NUMERIC  / power(10, t_in.decimals))  AS amount_in_human,
  (t.amount_out::NUMERIC / power(10, t_out.decimals)) AS amount_out_human
FROM trades t
LEFT JOIN tokens t_in  ON t_in.address  = t.token_in_address
LEFT JOIN tokens t_out ON t_out.address = t.token_out_address;
